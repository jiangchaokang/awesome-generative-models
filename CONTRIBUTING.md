# Contributing to awesome-generative-models

Thank you for contributing.

This repository is not a dump of papers.
It is a **curated, high-precision, English-only catalog** of recent generative-model work.

## Non-negotiable Rules

1. **English only**
   - Titles, summaries, notes, and page copy must all be in English.
   - No Chinese summaries.
   - Every entry must have a **1–3 sentence English summary**.

2. **Exact links only**
   - Never use search-result URLs as public `paper`, `repo`, or `homepage` links.
   - That means no Google Scholar search links, no GitHub search links, no generic search pages.
   - If a link is not confirmed, leave it empty.

3. **2025-01-01+ scope**
   - By default, only works **published/created on or after 2025-01-01** are allowed.
   - This floor is enforced automatically by `scripts/fetch_candidates.py` (using arXiv `published` date and GitHub repo `created_at`, never `updated_at`) so that an old project touched by a routine commit can never resurface as a "new" candidate.
   - Older works may be kept **only if** they were clearly active in 2025+ and you add:
     - `active_since`
     - `scope_note`

4. **One work, one primary artifact**
   - Every work appears in exactly one top-level artifact file.
   - Cross-cutting views must be expressed via tags (`domain`, `method`, `representation`, `conditioning`, `orgs`) and generated pages.

5. **Human review before merge**
   - Daily candidate harvesting is automatic.
   - Final inclusion in `data/*.jsonl` is always manual.

## Source Files You May Edit

Edit only the source data and docs:

- `data/image_2d.jsonl`
- `data/video.jsonl`
- `data/object_3d.jsonl`
- `data/scene_3d.jsonl`
- `data/world_4d.jsonl`
- `docs/taxonomy.md` (only when taxonomy must be expanded)

Do **not** manually edit generated files:

- `README.md`
- `00-surveys-and-foundations/**`
- `10-image-2d/**`
- `20-video/**`
- `30-3d-object-asset/**`
- `40-3d-scene/**`
- `50-4d-dynamic-scene-world/**`
- `90-topics/**`
- `91-organizations/**`

These are generated by `python scripts/build.py`.

## Classification Rules

### Rule 1 — Classify by final artifact

- Final output is a single image → `image_2d.jsonl`
- Final output is a video → `video.jsonl`
- Final output is a reusable 3D asset/object/avatar → `object_3d.jsonl`
- Final output is a multi-object 3D scene → `scene_3d.jsonl`
- Final output is a dynamic 3D/4D world / world model / simulation → `world_4d.jsonl`

### Rule 2 — `domain` is never a top-level directory

Examples:
- `autonomous-driving`
- `indoor`
- `human-avatar`
- `gameplay`

These are tags / generated topic pages only.

### Rule 3 — `orgs` is metadata, not taxonomy

Use `orgs` for:
- company badges
- university badges
- organization pages

But never use `orgs` to decide the primary artifact.

### Rule 4 — No link fabrication

If you cannot confirm the exact repo or paper URL, do **not** invent one.
Leave the field empty.

## Local Workflow

```bash
# 1) Fast validation: schema, vocab, dedupe, 2025+ scope -- no network, runs on every push
python scripts/validate_data.py --skip-network

# 1b) Optional deep validation: also verifies every exact link is alive and
#     refreshes GitHub star/license/archived cache (slower, network-bound;
#     this is what the weekly-link-check workflow runs)
python scripts/validate_data.py --write-cache

# 2) Rebuild generated pages
python scripts/build.py

# 3) Optional: refresh candidate inbox
python scripts/fetch_candidates.py --days 14

# 4) Commit
git add data docs metadata README.md 00-surveys-and-foundations 10-image-2d 20-video 30-3d-object-asset 40-3d-scene 50-4d-dynamic-scene-world 90-topics 91-organizations
git commit -m "data: update catalog"
git push
```

## Candidate Inbox Workflow

All automatically harvested materials are written to:

- `metadata/candidates/latest.md`
- `metadata/candidates/latest.json`
- `metadata/candidates/YYYY-MM-DD/`
- `metadata/candidates/seen_index.json` — long-term memory of every candidate ever surfaced

That dated folder contains raw harvested JSON and the Markdown review report.
Please review there first, then manually curate `data/*.jsonl`.

A few properties make this inbox trustworthy instead of noisy:

- **Delta, not re-scan.** Once a candidate is reported, it is recorded in
  `seen_index.json` and will not be reported again (it may resurface after
  `SEEN_TTL_DAYS` for one more look). A day with 0 new candidates is normal
  and expected — it means nothing genuinely new cleared the bar, not that the
  pipeline is broken.
- **True novelty, not "recently touched".** arXiv candidates are gated on the
  paper's `published` date; GitHub candidates are gated on the repo's
  `created_at`. A trivial commit to a years-old repository can never make it
  look like new work.
- **Hard 2025-01-01+ floor.** Enforced unconditionally, independent of the
  `--days` lookback window.
- **Quality gate.** Each candidate is scored and must clear `MIN_SCORE_ARXIV`
  / `MIN_SCORE_GITHUB` in `scripts/fetch_candidates.py` before it is even
  written to disk.
- **Rolling retention.** Raw dated folders older than `RETENTION_DAYS` (30)
  are pruned automatically; `seen_index.json` retains the dedupe memory.

## Common Rejection Reasons

An entry will be rejected if it has any of the following:

- Non-English summary
- Missing summary
- Wrong artifact classification
- Search-result URL used as a public link
- `open_source: true` without an exact repo
- Pre-2025 work without `active_since` and `scope_note`
- Unknown task / tag vocabulary
- Duplicate title / duplicate paper URL / duplicate repo URL
